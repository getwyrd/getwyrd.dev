Conformance vectors

Normative specification. Status: DRAFT — tracks chunk-format v1 (v0, unstable). This document uses RFC 2119 keywords: MUST, SHOULD, MAY.

Test vectors that any implementation of the on-disk format (../chunk-format/v1.md, decided in ADR-0019) MUST satisfy. They live with the spec, in the repository, so the spec and the reference implementation stay honest with each other mechanically: the chunk-format crate is run against these vectors in CI (cargo xtask conformance).

Layout

The first valid and invalid vectors are landed and wired into CI as of Milestone 0.2 (proposal 0001). They are produced deterministically by cargo xtask gen-vectors and committed to the repository; CI runs the read-only cargo xtask conformance check, which the chunk-format reference reader must pass, on every pull request.

conformance/
  vectors/
    v1/
      <name>.fragment      # raw bytes of a valid fragment
      <name>.expected.json # the parsed, verified interpretation a conforming reader must produce
  invalid/
    v1/
      <name>.fragment      # malformed fragments a conforming reader MUST reject
      <name>.reason.txt    # why it must be rejected

.fragment

Raw fragment bytes, exactly as a D server stores them. They are stored verbatim (marked binary in .gitattributes) so they round-trip byte-for-byte on every platform — the bytes are the contract.

.expected.json

The interpretation a conforming reader MUST produce for the sibling .fragment. To stay language-neutral and lossless, chunk_id and payload_length are decimal strings (both exceed JSON's safe-integer range) and the payload is lowercase hex:

field JSON type notes
format_version number u16
flags number u8
checksum_algo string crc32c | blake3
encryption_scheme string none
ec_scheme_type string none | replication | reed-solomon
ec_k number u8
ec_m number u8
ec_fragment_index number u16
chunk_id string decimal u128
payload_length string decimal u64
payload_hex string lowercase hex of the stored payload

.reason.txt

The first line MUST be error: <Variant>, naming the reference reader's rejection kind (e.g. error: BadMagic), so a vector cannot pass merely by being rejected for the wrong reason. Any following lines are human-readable prose.

Rules

  • A conforming reader MUST parse and verify every vector under vectors/ to match its .expected.json, and MUST reject every fragment under invalid/ with the error named on the first line of its .reason.txt.
  • A conforming writer MUST produce fragments the reference reader accepts.
  • Adding or changing a vector is part of the strict spec change process (a format version bump). Vectors are append-only within a format version.

These vectors are also how a future non-Rust client (ADR-0002, architecture section 8.6) proves it reads and writes data compatibly, rather than relying on reverse-engineering the reference code.
