Chunk / fragment on-disk format — v1

Normative specification. Status: DRAFT (v0, unstable — will change). This is the one normative, spec-first artifact in the project (ADR-0002). The byte layout is now fully specified and the reference implementation is built against it, but it is published as a draft with an explicit instability marker and is stamped v1 only once validated by a second independent reader implementation or a sustained fault-injection run (the rule in "Versioning" below).

This document uses RFC 2119 keywords: MUST, SHOULD, MAY.

The decisions behind this layout — u128 chunk IDs, the self-describing header length as the evolution mechanism, and the checksum-over-stored-bytes rule that keeps integrity verifiable without decryption — are recorded in ADR-0019.

Why this is specified

Data outlives software. A provider with petabytes written under format v1 MUST be able to read it with software written years later. This format is therefore the migration anchor and the stable contract that any reader implementation — the Rust client, a future migration tool, a non-Rust SDK — MUST honor. The on-disk bytes are the source of truth; software is replaceable around them.

Scope

This spec covers the on-disk/on-wire layout of a stored fragment: the bytes a D server persists and returns for a given chunk fragment. It does not cover metadata-store schemas, wire RPC, or placement — those are implementation-first (ADR-0002).

Conventions (normative)

  • All multi-byte integers are little-endian. A reader MUST interpret them as little-endian; there is no big-endian variant.
  • All offsets and sizes in this document are in bytes.
  • The header is a fixed core followed by an optional extension region. The core is the fields at offsets 0–43. The total header size (core + extension) is carried in header_length, so a reader locates the payload from the header alone, without understanding every field.

Fragment structure (normative)

A stored fragment MUST consist of a header, a payload, and a payload checksum, in that order:

Offset  Size  Field               Notes
------  ----  ------------------  --------------------------------------------
0       4     magic               MUST be 0x57595244 (ASCII "WYRD").
4       2     format_version      u16. The format version. A reader MUST reject
                                  a fragment whose version it does not
                                  understand rather than guess. v1 = 1.
6       2     header_length       u16. Total header size in bytes, core plus
                                  extension. In v1 with no extension this is 44.
                                  The payload begins at this offset.
8       1     flags               u8. Bit 0 (LSB) = payload is encrypted. All
                                  other bits reserved and MUST be 0 in v1.
9       1     checksum_algo       u8. Payload-checksum algorithm.
                                  0 = crc32c (default). 1 = blake3 (reserved;
                                  not produced by a v1 writer). Other values
                                  reserved.
10      1     encryption_scheme   u8. 0 = none (v1). AEAD schemes are reserved;
                                  a v1 writer MUST write 0 and MUST clear
                                  flags bit 0.
11      1     ec_scheme_type      u8. 0 = none/single-copy, 1 = replication,
                                  2 = reed-solomon. Other values reserved.
12      1     ec_k                u8. Data-fragment count k. 1 for none and for
                                  replication.
13      1     ec_m                u8. Parity-fragment count m. 0 for none and
                                  for replication.
14      2     ec_fragment_index   u16. This fragment's index within its stripe,
                                  0-based. Always 0 for none; the replica or
                                  parity index otherwise.
16      16    chunk_id            u128. The chunk identifier. MAY be generated
                                  without central coordination.
32      8     payload_length      u64. Length in bytes of the stored payload.
40      4     header_checksum     u32. crc32c over bytes [0, header_length),
                                  computed with this field's own four bytes
                                  taken as zero. Guards the header itself.
44      E     header_extension    E = header_length - 44. Empty in v1 (E = 0).
                                  Reserved for future fields (e.g. encryption
                                  nonce and key identifier, compression info).
44+E    N     payload             N = payload_length. The stored bytes
                                  (ciphertext if flags bit 0 is set, otherwise
                                  plaintext).
44+E+N  V     payload_checksum    Per checksum_algo: 4 bytes for crc32c, 32
                                  bytes for blake3. Computed over the stored
                                  payload bytes (see "Checksums").

The v1 core header is therefore 44 bytes. The payload starts at header_length (44 when there is no extension).

Field rules (normative)

  • A reader MUST verify magic and reject any fragment that does not begin with it.
  • A reader MUST reject a fragment whose format_version it does not understand, rather than attempt to parse it.
  • A reader MUST read header_length to locate the payload and MUST treat any bytes between offset 44 and header_length as a header extension it MAY not understand. A reader that does not understand an extension MUST skip it (using header_length) and MUST NOT guess its contents. A reader MUST NOT fail solely because an extension is present, except where a set flags bit (e.g. encryption) means the reader cannot correctly return the payload — in which case it MUST refuse rather than return undecrypted or misinterpreted bytes.
  • A v1 writer MUST write flags = 0, encryption_scheme = 0, and an empty extension (header_length = 44).
  • The ec_scheme_type, ec_k, ec_m, and ec_fragment_index recorded here MUST agree with the chunk's authoritative metadata record. The header is self-describing so a fragment validates standalone (a scrubber needs no external system); the metadata is authoritative so the system knows a chunk's scheme without reading fragments. A disagreement is a detectable corruption.

Checksums (normative)

  • The header checksum (header_checksum) is crc32c computed over the header bytes [0, header_length) with the four bytes of the header_checksum field itself treated as zero during computation. A reader MUST verify the header checksum before acting on any other header field.
  • The payload checksum (payload_checksum) is computed over the stored payload bytes — that is, over the ciphertext when the payload is encrypted, and over the plaintext when it is not. This is mandatory so that integrity is verifiable without the decryption key: a D server or scrubber holding only ciphertext MUST be able to detect bit-rot and corruption without decrypting. A reader MUST verify the payload checksum before returning data, and a scrubber MUST verify it in the background (architecture section 6.3).
  • Authenticity of the plaintext under encryption is the responsibility of the AEAD scheme (recorded by a future encryption_scheme), and is separate from this storage-integrity checksum.
  • The algorithm is identified by checksum_algo. A v1 writer MUST use crc32c (0). blake3 (1) is reserved for future content-addressing or cryptographic-integrity use and MUST NOT be produced by a v1 writer; a reader MAY support it.

Erasure-coding scheme, recorded per chunk (normative)

The header records the EC scheme per fragment (ec_scheme_type, ec_k, ec_m, ec_fragment_index) so that a zone which grows from replication(n) into rs(k,m) carries mixed-era data correctly (ADR-0008): old chunks remain readable under their original scheme while new chunks use the new one. A reader MUST use the scheme recorded in the fragment (cross-checked against the chunk metadata) and MUST NOT assume a single zone-wide scheme.

Versioning (normative)

  • Every fragment MUST carry its format_version.
  • A reader MUST reject a fragment whose version it does not understand rather than guess.
  • Backward-compatible additions (new optional header-extension fields, new reserved code points for checksum_algo, encryption_scheme, or ec_scheme_type) MAY be made without a version increment, because a conforming reader skips extensions it does not understand and rejects code points it does not recognize. Such additions MUST be accompanied by new conformance vectors.
  • A change that alters the meaning or layout of an existing core field, or that a conforming v1 reader could misinterpret, is a breaking change and MUST increment format_version. A version increment is an ecosystem event and follows the strict change process (this document plus updated conformance vectors).
  • This format remains v0/unstable until validated by a second independent reader implementation or a sustained fault-injection run, at which point it is stamped v1. Until then, the layout MAY change.

Reserved for future use (informative)

The following are reserved now so they can be added without a breaking change:

  • Encryption — flags bit 0, the encryption_scheme byte, and the header extension region (for nonce and key identifier). Client-side envelope encryption is a planned future feature (ADR-0005 §8.5); the checksum-over-stored-bytes rule above already accommodates it.
  • Content addressing / cryptographic integrity — checksum_algo = 1 (blake3).
  • Compression — a future flags bit and a scheme byte, with any parameters in the header extension.

Conformance

A conforming reader MUST correctly parse and verify every vector in ../conformance/. A conforming writer MUST produce fragments that the reference reader accepts. See ../conformance/v1.md.
