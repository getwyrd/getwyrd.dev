Specification

Chunk / fragment on-disk format — v1

Status: DRAFT (v0, unstable — will change). This is the one normative, spec-first artifact in the project (ADR-0002). It is published as a draft with an explicit instability marker; the reference implementation is built against it, and it is stamped v1 only once validated by a second reader implementation or a sustained fault-injection run.

This document uses RFC 2119 keywords: MUST, SHOULD, MAY.

Why this is specified

Data outlives software. A provider with petabytes written under format v1 MUST be able to read it with software written years later. This format is therefore the migration anchor and the stable contract that any reader implementation — the Rust client, a future migration tool, a non-Rust SDK — MUST honor. The on-disk bytes are the source of truth; software is replaceable around them.

Scope

This spec covers the on-disk/on-wire layout of a stored fragment: the bytes a D server persists and returns for a given chunk fragment. It does not cover metadata-store schemas, wire RPC, or placement — those are implementation-first (ADR-0002).

Fragment structure (normative — to be completed)

A stored fragment MUST consist of a header followed by a payload. The header MUST allow a reader to determine, without consulting any external system:

  • A magic number identifying the format.
  • The format version (so mixed-era data is readable — see below).
  • The erasure-coding scheme in effect for this fragment's chunk, recorded per chunk so a zone that grows from replication(n) into rs(k,m) carries mixed-era data correctly (ADR-0008). This MUST include the scheme identifier and the fragment's index within the stripe.
  • The chunk identifier.
  • A checksum of the payload (algorithm identified in the header), which a reader MUST verify before returning data and a scrubber MUST verify in the background (architecture section 6.3).
  • The payload length.

The exact byte layout, field widths, endianness, checksum algorithm identifiers, and EC scheme identifiers are [TO BE SPECIFIED] during Milestone 0–1 implementation and recorded here before v1 is stamped.

Versioning rules (normative)

  • Every fragment MUST carry its format version.
  • A reader MUST reject a fragment whose version it does not understand, rather than guess.
  • A version increment is a breaking ecosystem event and follows the strict change process (this document, plus updated conformance vectors).

Conformance

A conforming reader MUST correctly parse and verify every vector in ../conformance/. A conforming writer MUST produce fragments that the reference reader accepts. See ../conformance/README.md.
